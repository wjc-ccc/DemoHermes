## 需求 1：构建 Gateway 类、message 构建函数、message bus 输入输出、session key 构建与上下文寻找，优化 gateway 目录结构

### 尝试与反馈

1. **先盘点现状**：`gateway/` 下 `gateway.py` 和 `__init__.py` 是空文件，真正的 Gateway 实现躺在 `runner.py` 里，`queues.py` 是双队列，`session_store.py` 同时有 JSONL 落盘 + 内存两种实现。
   → 反馈：命名与内容不符（空壳文件 + 实现错位）、JSONL 落盘在当前阶段是冗余（你也明确指出 session 只用 dict）。
2. **message bus 的取舍**：考虑过是否按 7.14 文档 FAQ 保持 `GatewayQueues` 原名小改。
   → 反馈：`queues.py` 名字太泛，看不出"这是 Channel⇄Gateway 的传送带"；且缺独立的"总线"抽象，输入输出方法散落在队列原语上。
3. **session 存储的取舍**：保留 ABC + JSONL + Memory 三件套，还是砍到只剩 dict。
   → 反馈：按你的要求"现在只用 dict 即可，保证代码干净不冗余"，ABC 和 JSONL 全删，接口压缩为 4 个方法。
4. **message 构建函数放哪**：一度想放 `gateway/message_factory.py`（你把它列在 gateway 能力里）。
   → 反馈：assistant/tool 消息的构建者是 Loop 而不是 Gateway，放 gateway 会造成 core→gateway 的反向依赖。

### 最终做法

```
agent/gateway/
├── __init__.py        # 统一导出（Gateway / MessageBus / DictSessionStore / build_session_key / FrontierHttpServer）
├── gateway.py         # Gateway 类：channel 注册、submit/ask、session_key 串行、斜杠命令短路、出站分发、status 观测
├── message_bus.py     # MessageBus：publish_inbound/next_inbound/publish_outbound/next_outbound，None 哨兵关闭
├── session_key.py     # build_session_key（保持原实现，全项目唯一公式）
├── session_store.py   # DictSessionStore：find / get_or_create / save / reset / list_sessions（纯 dict + RLock）
└── http_api.py        # FrontierHttpServer：纯标准库 HTTP/SSE（见需求 4）
```

- **message 构建函数**落在 `agent/core/messages.py`（`build_user_message / build_assistant_message / build_tool_message`），`Gateway.build_message()` 委托调用——gateway 拥有该能力，但实现放在依赖方向正确的一层。
- **上下文寻找**：`build_session_key(source)` → `DictSessionStore.get_or_create()`，同 key 一把锁串行（`_lock_for`），不同 key 线程池并行。
- **删除**：`runner.py`、`queues.py`、JSONL/Memory 双实现、`GatewayRunner` 兼容别名。全仓 grep 确认无残留引用。
- 全部公共函数带中文 docstring/注释。

## 需求 2：优化 AgentLoop——合理访问各类 provider、多步解析大模型输入输出（工具调用、skill 调用），构建测试 tool 和 skill

### 尝试与反馈

1. **provider 访问**：原 `loop.py` 直接 `AgentLoop(self.model)` 吃一个写死的 DeepSeekModel；`model/registry.py` 是空壳。
   → 反馈：实现 ModelRegistry（register/get/default），Loop 支持"直传 model"与"注册表按名解析"两种方式，新增 provider 不改 Loop。
2. **真实密钥踩坑（重要）**：联调时 DeepSeek 端点 401。排查发现 `.env` 里的密钥是**火山引擎 Ark 密钥**（`ark` 前缀），对 `api.deepseek.com` 和 Ark 的 OpenAI 兼容端点 `ark.cn-beijing.volces.com/api/v3` 都 401；最终发现它对 `.env` 里 `ANTHROPIC_BASE_URL` 指向的 **Anthropic 兼容端点**（`.../api/plan/v1/messages`）返回 200。
   → 反馈：只实现 OpenAI 协议的 provider 在当前密钥下根本跑不通，必须实现 Anthropic 协议。
3. **Anthropic 协议适配**：消息格式双向转换（system 抽顶层、assistant tool_calls→tool_use 块、相邻 tool 消息合并成一条 user 消息的 tool_result 块、input_schema 转换）。
   → 反馈：用 httpx 直连实现 `ClaudeModel`（不引 anthropic SDK，venv 里没装），4 个纯本地单元测试覆盖转换逻辑。`ModelRegistry.default()` 检测 `.env` 显式配置了 `ANTHROPIC_BASE_URL` 时默认用 claude。
4. **多步解析**：Loop 升级为 think⇄act 内循环——模型返回 tool_calls 就执行、回注、再调，直到纯文本或触达 `max_iterations`。
   → 反馈：OpenAI/Anthropic 两家的 tool 协议差异由 ContextBuilder 的消息映射 + 各 Model 的响应解析分别消化，Loop 骨架不变（符合 7.14 文档契约）。
5. **skill 调用怎么实现**：考虑过给 Loop 单开 skill 分支。
   → 反馈：更干净的做法是"技能调用复用工具调用链路"——`use_skill` 就是一个普通 Tool，Level 1 摘要注入 system prompt 引导模型调用它加载 Level 3 正文。Loop 零改动支持 skill。
6. **测试 tool/skill 选型**：calculator（AST 白名单安全求值，不用 eval）+ weekly_report（SKILL.md：frontmatter 元信息 + 正文步骤/格式模板）。
   → 反馈：calculator 验证了参数解析与安全拦截（`__import__('os')` 注入被拒）；weekly_report 实测中模型主动调用 use_skill 并严格按模板输出周报。

### 最终做法

- **Loop 容错兜底**：模型调用失败重试（退避 0.5s×n，默认 2 次）→ 仍失败返回 `TurnResult(error)`；工具异常由 `ToolRegistry.dispatch` 兜底成 `ToolResult(ok=False)` 回注让模型自我修正；超迭代写兜底文案并 `error="max_iterations"`；全程 logging + EventBus 事件。
- **实测记录**：普通对话 ✅、`(123+456)*7` 再开根号 → calculator 调用 ✅、"帮我写周报" → use_skill 调用 + 模板化输出 ✅。

## 需求 3：构建风格清晰简明的前端界面（frontier/），agent loop 可视化

### 尝试与反馈

1. **技术选型**：考虑过 FastAPI（venv 没装）、引入 npm 构建（过重）。
   → 反馈：venv 里只有 uvicorn 没有 ASGI 框架；最终**后端纯标准库** `ThreadingHTTPServer`（POST /api/chat + SSE /api/events + /api/status /api/sessions /api/reset），**前端纯静态三件套**（index.html + style.css + app.js，无构建步骤）。零新增依赖。
2. **实时可视化方案**：WebSocket 需要第三方库 → 改 SSE（浏览器原生 EventSource，自动重连），事件源就是 Loop 里发的 EventBus 事件。
   → 反馈：SSE 单向推送完全够用，且与 Gateway 的线程模型天然匹配（一个连接一个线程）。
3. **Windows 中文编码坑**：curl 测试时 GBK 字节导致 `UnicodeDecodeError`；CLI 管道输入出现 surrogate 编码错误。
   → 反馈：`_read_json_body` 做 UTF-8→GBK 解码兜底；CLI 模式在非 tty 时 reconfigure stdin/stdout 为 UTF-8（交互终端保持系统编码不动）。
4. **CLI 重复打印**：ask() 返回值手动打印 + Gateway 出站分发 deliver 打印 → 一句话回两遍。
   → 反馈：CLI 模式注册 `quiet=True` 的 CliChannel，只由 ask 返回处打印一次。

### 最终做法

- **布局**：顶栏（连接指示灯 + model/tools/skills 徽标 + 新会话按钮）；左侧对话区（用户/AI/错误气泡）；右侧 **Agent Loop 可视化面板**——SSE 事件按 turn_id 分组为轮次卡片，逐步展示：▶ 开始 → ↑ 调用模型（迭代次数/上下文规模）→ ↓ 模型返回（文本预览/工具调用）→ 🔧 工具调用 → ✔ 工具结果（耗时）→ 🏁 完成。
- **启动**：根目录 `python main.py`（HTTP 模式 :8000，或 `--cli` 终端模式）；`python frontier/main.py`（:5173，自动开浏览器）。CORS 已放行。

## 需求 4（总结性）：gateway / message bus / channel 关系、去冗余、一键启动、开发流程

### 最终做法

1. **三者关系**：`Channel.parse_to_InboundEvent` → `Gateway.submit/ask` → `MessageBus.in_dto` → worker（session_key 加锁 → store 找上下文 → Loop）→ `MessageBus.out_dto` → 按 `source.channel` 路由 → `Channel.deliver_to_OutboundEvent`。`BaseChannel` 修复了两个 bug：status 从**类属性共享 dict** 改为实例级；新增 `bind_gateway` 使 `send_gateway` 真正可用。
2. **去冗余**：删 `runner.py`/`queues.py`/JSONL store/空壳 `gateway.py` 内容错位；session 存储只剩 dict；`test/test_sending.py`（坏导入）替换为 31 项 pytest。
3. **测试**：31 项全过（session_key 公式 / dict store / 消息工厂 / calculator 安全 / skill 加载 / 上下文协议映射 / 多步 Loop / 三类容错 / Gateway 总线回路 / 同 key 串行 / Claude 协议转换）。真实模型端到端实测通过。
4. **开发流程**：新建分支 `feature/gateway-loop-frontend` → 先快照你的在途改动再开发 → 本地 pytest 全过 + 双进程联调实测 → 推送分支 → 创建 PR #1（gh CLI 缺失，改用 git 凭据管理器中的 token 调 GitHub REST API）→ **CI 说明：仓库未配置 CI 工作流，验收以本地 31 项 pytest + 真实链路联调为准** → 合并入 main。

## 附：已知边界 / 后续建议

- DictSessionStore 进程重启即清空（当前预期行为）；需要持久化时按同接口换 SQLite。
- `.env` 的密钥命名有误导性（Ark 密钥放在 `DEEPSEEK_API_KEY`），ClaudeModel 做了 `ANTHROPIC_API_KEY → DEEPSEEK_API_KEY` 的回退读取；建议后续把密钥挪到 `ANTHROPIC_API_KEY`。
- `test/` 在 .gitignore 中（你的既有约定），测试不随仓库分发；如需 CI 跑测试，先把测试目录移出 ignore 并加 workflow。
- 前端是定死 `127.0.0.1:8000` 的后端地址；跨机部署时抽成配置。
