> 目标：定义 DemoCursor 中 **Agent Loop 应具备的能力**、**如何与多 Channel 关联/隔离**、以及 **Session 存储与恢复契约**。

> 本文为架构方案，不直接改业务代码；实现顺序见文末。

## 一、背景与结论

Hermes 可粗分为：

| 部分 | Hermes 对应 | DemoCursor 对应 |
|------|-------------|-----------------|
| Channel | `gateway/platforms/*` + CLI | `agent/channel/*` + `gateway/cli` / `api` |
| Orchestrator | `GatewayRunner` | 薄层 Gateway（`gateway/runner` 或并入 `core/session`） |
| Agent Loop | `AIAgent.run_conversation` / `conversation_loop.py` | `core/loop.AgentLoop.run_turn` |
| provider | `agent/transports/*` + client | `model/*`（`BaseModel.chat`） |
| Storage | `SessionStore` + `SessionDB` | Session index + TranscriptStore（后挂 memory） |

**一句话结论：**

- Loop 是核心：只做 `observe → think → act → persist`（及 tool 内循环），**不认识任何平台 SDK**。
- Channel（含 CLI / GUI / 飞书 / 微信）只负责 I/O 归一化与回写，**可无限扩展且不影响 Loop**。
- 多 Channel 隔离靠确定性 **`session_key`**，不靠 `Session.channel` 字符串。
- 存储分两层：`session_key → session_id` 索引 + transcript 正文；恢复按索引加载后再进 Loop。

从 Hermes **值得抄**的：单一入站 DTO、`session_key`、通道无关的返回值、ephemeral vs persisted、Adapter 内 busy/interrupt。  
**不抄**：`GatewayRunner` / `run_agent` 万行巨石、平台特例渗进编排层。

## 二、分层与目录映射

```
外部世界（用户 / 飞书 / 微信 / 终端）
        │
        ▼
┌─────────────────── Channel ───────────────────┐
│  cli / api / wechat / feishu / …              │
│  平台协议 → InboundEvent                      │
│  OutboundReply → 平台协议                     │
│  busy 队列 / typing / 鉴权（平台侧）留在这里   │
└───────────────────┬───────────────────────────┘
                    │ InboundEvent
                    ▼
┌─────────────────── Gateway（薄编排）──────────┐
│  build_session_key(source)                    │
│  SessionStore.get_or_create / load / reset    │
│  AgentLoop.run_turn(...)                      │
│  按 source 选中 adapter.send(OutboundReply)   │
└───────────────────┬───────────────────────────┘
                    │ Session + Message
                    ▼
┌─────────────────── Core Loop ─────────────────┐
│  observe → think → act → persist              │
│  think/act 内可嵌套：API ↔ tools 直到无调用   │
│  ContextBuilder / ToolRegistry / provider     │
└───────────────────┬───────────────────────────┘
                    │ TurnResult
                    ▼
              回到 Gateway → Channel
```

| 层 | 模块路径 | 职责 |
|----|----------|------|
| Channel | `agent/channel/*`，`agent/gateway/cli.py` / `api.py` | 入站归一化、出站投递；CLI/API 也是 Channel |
| Gateway | 建议 `agent/gateway/runner.py` | resolve session → 调 loop → 回写 |
| Core Loop | `agent/core/loop.py` | OTAP + tool 内循环 |
| Session | `agent/core/session.py` + Store | 生命周期、index、transcript |
| Context | `agent/core/context_builder.py` | Session → OpenAI `messages` |
| provider | `agent/model/*` | LLM inference |
| Storage | `data/sessions/`（后挂 `memory/`） | index + transcript；记忆旁路 |
| Evolution | `agent/evolution/*` | persist 之后旁路，不改 `run_turn` 签名 |

**禁止：** Loop / ContextBuilder / Model 中出现微信/飞书 SDK、webhook、素材下载逻辑。

## 三、Loop 能力清单

按阶段标注。A 是今日设计后**下一步实现**目标；B–E 为契约预留。

### A. 最小可跑（阶段 1）

| # | 能力 | 说明 |
|---|------|------|
| 1 | Observe | `InboundEvent` → `Message(role=user)` → 写入 `Session.messages` |
| 2 | Think | `ContextBuilder` 组 OpenAI 格式 → `model.chat(messages)` |
| 3 | Act（无 tool） | 回复 → `Message(role=assistant)` 写入 Session |
| 4 | Persist（最小） | 本 turn 新增消息 flush 到 TranscriptStore（可先不写 memory） |
| 5 | TurnResult | 返回 `final_text` / `completed` / `error`，供 Channel 回写 |

### B. 工具循环（阶段 2，今日只定契约）

| # | 能力 | 说明 |
|---|------|------|
| 6 | 内循环 | `while tool_calls and iterations < max`：执行 → 回注 → 再 think |
| 7 | Tool 回注 | 为 `Message.role` 增加 `"tool"`（实现阶段再改 types）；ContextBuilder 负责映射 API 格式；**Loop 骨架不变** |
| 8 | 预算与中断 | `max_iterations`；`interrupt` 由 Channel/Gateway 设 flag，Loop 每轮检查 |

### C. 与 Channel / 多会话相关（本方案重点）

| # | 能力 | 说明 |
|---|------|------|
| 9 | 通道无关入口 | `run_turn(session, user_message, *, ephemeral_prompt=None) -> TurnResult` |
| 10 | 不持有 Channel | Loop 不出站；Gateway 用 `TurnResult` + `SessionSource` 调 `adapter.send` |
| 11 | Ephemeral vs Persisted | `channel_prompt`（「你在飞书群里」）仅注入当次 API，**不写入** transcript |
| 12 | 单 session 串行 | 同一 `session_key` 同时只跑一个 turn；排队/打断在 Channel/Gateway |

### D. Storage / Restore

| # | 能力 | 说明 |
|---|------|------|
| 13 | 双层 ID | `session_key`（逻辑槽）与 `session_id`（某次 transcript UUID）分离 |
| 14 | Restore | key → index → load messages → 重建 Session →（可选）冻结 system prompt |
| 15 | Persist 语义 | 只 append 未 flush 后缀；避免重复写 |
| 16 | Reset | idle / daily / `/new` → 新 `session_id`；旧 transcript 保留可检索 |

### E. 后续挂载点（不改 Loop 签名）

| # | 挂载点 | 说明 |
|---|--------|------|
| 17 | Events | `turn_start` / `llm_response` / `tool_call` → 可视化 |
| 18 | Memory L1 | persist 后写 USER.md / MEMORY.md（按 **user** 维度，非 channel） |
| 19 | Trajectory | 可回放执行轨迹 |
| 20 | Evolution | reviewer 旁路，不阻塞主循环 |

## 四、类型契约

在现有 [`agent/core/types.py`](../../agent/core/types.py)（`Message` / `Session` / `ToolCall`）之上，实现时新增下列类型（可落在 `types.py` 或 `gateway/types.py`，以「Channel 层可见、Loop 不依赖平台」为准）。

### 4.1 SessionSource — 消息从哪来、回哪去

```python
class SessionSource(BaseModel):
    channel: Literal["cli", "api", "wechat", "feishu", ...]
    chat_id: str
    chat_type: Literal["dm", "group", "thread"] = "dm"
    user_id: str | None = None
    user_name: str | None = None
    thread_id: str | None = None
    reply_to_message_id: str | None = None
    metadata: dict = {}
```

用途：拼 `session_key`、路由回写、拼 ephemeral 平台上下文。

### 4.2 InboundEvent — Channel → Gateway 唯一入站 DTO

```python
class InboundEvent(BaseModel):
    text: str
    source: SessionSource
    content: list[ContentPart] = []      # 多模态后置
    channel_prompt: str | None = None    # ephemeral，不进 transcript
    metadata: dict = {}
```

### 4.3 OutboundReply — Gateway → Channel 出站

```python
class OutboundReply(BaseModel):
    text: str
    reply_to: str | None = None
    metadata: dict = {}
```

### 4.4 TurnResult — Loop → Gateway 唯一出站契约

```python
class TurnResult(BaseModel):
    final_text: str | None = None
    session_id: str
    completed: bool = True
    interrupted: bool = False
    error: str | None = None
    # 可选：本 turn 新增 messages / tool_calls，供观测
```

CLI、微信、飞书、HTTP API **都只吃/吐这些 DTO**，与 Hermes 的 `MessageEvent` + `run_conversation` 返回 dict 同构。

### 4.5 与现有 Session 的约定

- 保留 `Session.channel` 作展示/调试字段。
- **真正隔离与路由用 `session_key`**，存在 SessionEntry（index）或 `Session.metadata["session_key"]`，禁止仅用 `"wechat"` 字符串当唯一键。
- `Session.author_id` ↔ `SessionSource.user_id`。
- 实现阶段为 `Session` 补 `add_message()`：维护 `seq_id`、`message_count`、`updated_at` / `last_active_at`。

## 五、多 Channel 关联与隔离

### 5.1 如何「关联」Channel

关联 = **回得去 + 带得上上下文**，不是把 Channel 对象塞进 Loop。

```
InboundEvent.source  ──►  session_key / session_id
                     ──►  ephemeral channel_prompt
TurnResult.final_text + source ──► adapter.send(chat_id, ...)
```

Gateway 持有 `adapters: dict[channel, BaseChannel]`；根据 `source.channel` 选型回写。Loop 始终看不到 adapter。

### 5.2 如何「分离」多个 Channel

| 隔离维度 | 机制 |
|----------|------|
| I/O | 每平台一个 Adapter；统一产出/消费 DTO |
| 会话 | `build_session_key(source)` 唯一实现点 |
| 并发 | 按 `session_key` 串行；不同 key 可并行 |
| Prompt | 平台差异进 ephemeral，不全量入 history |
| 进程内共享 | 多个 Adapter 可共用同一 SessionStore / 同一 Model 实例 |

同一用户在微信与飞书是**两个 session_key**（除非显式做「账号绑定合并」，属后置需求，本方案不做）。

### 5.3 session_key 公式（确定性，全项目唯一）

```text
DM:
  agent:{agent_id}:{channel}:dm:{chat_id}[:thread_id]

Group（默认每用户独立会话）:
  agent:{agent_id}:{channel}:group:{chat_id}:user:{user_id}[:thread_id]

Thread 默认：thread 内共享（不强制再拼 user），与 Hermes 一致；
若需要 thread 内按用户隔离，再追加 :user:{user_id}（配置开关，默认关）。
```

示例：

```text
agent:main:cli:dm:local
agent:main:wechat:dm:wxid_abc
agent:main:feishu:group:oc_xxx:user:ou_yyy
```

`agent_id` 先固定为 `main`，便于日后多 Agent 共存。

### 5.4 BaseChannel 契约（示意）

```python
class BaseChannel(ABC):
    name: str  # "cli" | "wechat" | ...

    @abstractmethod
    async def start(self, on_event: Callable[[InboundEvent], Awaitable[None]]): ...

    @abstractmethod
    async def send(self, source: SessionSource, reply: OutboundReply) -> None: ...

    async def stop(self): ...
```

- `start`：把平台回调转成 `InboundEvent`，交给 Gateway handler。
- `send`：按 `source.chat_id` / `reply_to` 打回平台。
- busy / debounce / typing：实现放在具体 Adapter，不进 Loop。

## 六、端到端消息流

```
[飞书 / 微信 / CLI / API]
        │ 平台 SDK / input()
        ▼
BaseChannel：解码、（可选）鉴权、构图文
        │ 产出 InboundEvent(source=SessionSource(...))
        ▼
Gateway.handle(event)
        │ session_key = build_session_key(event.source)
        │ 同 key busy? → 排队或设 interrupt
        ▼
SessionStore.get_or_create(session_key)
        │ 评估 idle/daily reset → 可能新建 session_id
        │ load_transcript(session_id) → Session
        ▼
构造 Message(role=user, content_text=event.text, ...)
AgentLoop.run_turn(session, user_msg, ephemeral_prompt=event.channel_prompt)
        │ observe → think ↔ act(tools) → persist
        ▼
TurnResult
        ▼
Gateway → adapter.send(source, OutboundReply(text=final_text, reply_to=...))
        ▼
用户看到回复
```

CLI 与微信走**同一条路径**；差别只在 Adapter。

## 七、Loop 内部构建（OTAP + tool 内循环）

沿用 [`7.13 简单Loop实现.md`](./7.13%20简单Loop实现.md) 的四段命名，升级为可扩展骨架。

### 7.1 入口签名

```python
class AgentLoop:
    def __init__(self, model, context_builder_factory, tool_registry=None, store=None):
        self.model = model
        self.make_builder = context_builder_factory
        self.tools = tool_registry  # 阶段 1 可为 None
        self.store = store
        self.max_iterations = 20
        self._interrupt = False

    def request_interrupt(self) -> None:
        self._interrupt = True

    def run_turn(
        self,
        session: Session,
        user_message: Message,
        *,
        ephemeral_prompt: str | None = None,
    ) -> TurnResult:
        ...
```

### 7.2 伪代码

```text
run_turn(session, user_message, ephemeral_prompt=None):
    self._interrupt = False
    observe:
        session.add_message(user_message)

    api_calls = 0
    while api_calls < max_iterations:
        if self._interrupt:
            persist(); return TurnResult(interrupted=True, ...)

        think:
            msgs = ContextBuilder.build(session, ephemeral=ephemeral_prompt)
            resp = model.chat(msgs)   # 阶段 2：返回 content 和/或 tool_calls
            api_calls += 1

        act:
            if resp.has_tool_calls:
                append assistant(tool_calls) to session
                for each call: execute → append tool result message
                continue
            else:
                append assistant(content) to session
                break
    else:
        # 超过迭代：可写兜底文案或 error

    persist:
        store.append_new_messages(session)   # 只 flush 未写后缀
        # hooks: events / memory / trajectory → no-op 直至阶段接入

    return TurnResult(final_text=..., session_id=session.session_id, completed=True)
```

### 7.3 ContextBuilder 约定

阶段 1 已具备：system + 多轮对话。扩展顺序：

1. system（会话冻结快照优先）
2. ephemeral 平台说明（**仅 API 当次**，不落库）
3. hot memory / skills（阶段 3+）
4. `session.messages` → OpenAI 角色映射（含后续 `tool`）

### 7.4 与 Hermes 的对应

| Hermes | DemoCursor |
|--------|------------|
| `run_conversation` | `AgentLoop.run_turn` |
| `_execute_tool_calls` | act 内 tool 分支 |
| `_persist_session` | persist + TranscriptStore |
| `GatewayRunner._handle_message` | Gateway.handle |
| `MessageEvent` / `SessionSource` | `InboundEvent` / `SessionSource` |
| `ProviderTransport` | 现阶段 `BaseModel.chat`；协议分裂时再抽 Transport |

## 八、Storage 与 Restore

### 8.1 两层标识

| 概念 | 含义 | 存储 |
|------|------|------|
| `session_key` | 逻辑对话槽（平台+聊天+用户/线程策略） | index：`data/sessions/index.json` 或 SQLite 表 |
| `session_id` | 某次具体 transcript | `data/sessions/{session_id}.jsonl` 或 SQLite messages 表 |

为何分离：同一微信 DM 可因 `/new` 或 idle 复位产生多个 transcript，但逻辑槽不变，便于「继续聊」与「新开聊」并存。

### 8.2 SessionEntry（index 条目）

```text
session_key
session_id          # 当前活跃 transcript
channel / chat_id / user_id
created_at / last_active_at
system_prompt_snapshot   # 可选，会话开时冻结
reset_policy            # idle | daily | none
```

### 8.3 TranscriptStore

- `append_messages(session_id, messages: list[Message])` — 只写未 flush 后缀
- `load_messages(session_id) -> list[Message]`
- 内容为完整对话事实（含后续 tool 相关字段）；**不含** ephemeral `channel_prompt`

阶段 1 可用 JSONL；与 L2 episodic（SQLite+FTS）对齐时可迁到同一 DB，接口不变。

### 8.4 Restore 固定步骤

1. `session_key = build_session_key(source)`
2. 查 index；若 idle/daily 到期 → 创建新 `session_id`，更新 index
3. `messages = load_messages(session_id)` → 填入 `Session`
4. 取 `system_prompt_snapshot`；若无则现算并写入 index
5. `run_turn`；结束后更新 `last_active_at` 并 flush

### 8.5 Reset

| 触发 | 行为 |
|------|------|
| `/new` | 同 `session_key` 换新 `session_id` |
| idle 超时 | 自动换新 id，可选通知用户 |
| daily | 按自然日切 |
| `/resume <id>` | 后置：把 index 指回历史 session_id |

旧 transcript **不删**，供检索与进化。

### 8.6 Memory 与 Channel 的关系（预告）

- L1（USER.md / MEMORY.md）按 **user_id（人）** 维度，不按 channel：同一人在微信/飞书可共享画像。
- L2 episodic 可按 `session_id` 或 `user_id` 检索。
- Loop 的 persist 只保证 transcript；memory 通过 hook 异步写入，避免阻塞回复。

## 十二、FAQ：双队列 Gateway、字段解析、存储选型

### Q1：要不要独立 message bus 进程？

**不要另起 OS 进程。** 实现为**进程内双队列**（`in_dto` / `out_dto`）：

```text
Channel.parse_to_InboundEvent(raw)
      │
      ▼
  [in_dto Queue]  ←── 各 Channel 只 put，不碰 Session/Loop
      │
      ▼
 Gateway worker：build_session_key → 同 key 加锁串行
      → SessionStore.load → AgentLoop.run_turn → Store.save
      │
      ▼
  [out_dto Queue]
      │
      ▼
 Channel.deliver_to_OutboundEvent(OutboundReply)  ←── 按 source.channel 路由
```

多渠道「串行」= **同一 session_key 禁止并行两轮**；不同 key 可并行。  
DTO 与 Session/Message 互补：前者管运输，后者管对话事实。

### Q2：字段解析在哪？谁生成 session_key？

| 步骤 | 归属 |
|------|------|
| 微信 `content` → `text` | **Channel.parse_to_InboundEvent** |
| 拼 `SessionSource` | **Channel** |
| `build_session_key` / Store / 调 Loop | **Gateway** |
| 写 Message、调 model | **Loop** |

### Q3：Session 怎么存？JSONL vs Redis+SQL

| 方案 | 适用 | 说明 |
|------|------|------|
| **JSONL + index.json（当前默认）** | 单机开发 | `data/sessions/index.json` + `transcripts/{id}.jsonl` |
| **SQLite（推荐下一升级）** | 单机生产 | index+messages 同库，易 FTS，对齐 L2 |
| **Redis + SQLite/Postgres** | 多进程/多机 | Redis：热索引+分布式锁；SQL：transcript。队列可换 Redis List |

路径：JSONL → 阶段 3 前迁 SQLite → 多机再上 Redis。接口保持 `SessionStore` ABC。
