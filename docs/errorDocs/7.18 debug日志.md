### 1. message bus如何实现的？

- 本质两个queue，一个负责进入inbound queue 一个负责出 outbound queue，消费者只有gateway 生产者是各个channel
- 提供的功能就是`put、get、大小约束、结束信号、长度观测`。
- gateway负责实例构建 各个channel只需要能传入信息给gateway即可（后续需要做一些缓冲）

核心实现：一个queue模版类别，俩queue分别inbound和outbound 实现函数两个队列的存放和拿取（publish/next）

### 2. session key如何唯一构建？

**本质：取决于 agent是哪个 channel是哪里 个人还是群聊 对应的唯一id是什么**

- chat_type 实际有三个取值 dm / group / thread

```python
    """
    DM:    agent:{id}:{channel}:dm:{chat_id}[:thread:{tid}]
    Group: agent:{id}:{channel}:group:{chat_id}:user:{uid}[:thread:{tid}]
    """
```

### 3. gateway的职责是哪些？

`本质中转站 信息转换 + 上下文数据库/内存搜索 + agent loop构建`

- 处理message bus中的信息 包含不限于多线程 读取 存放 ，注册channel
- 构建唯一的session key 并且能够识别是否为新对话和一些斜杠指令根据key提取transcript组建为完整的session给到
- 把消息给到agent loop ，同session key 串行 / 异 key 并行`：每 key 一把锁 lockfor + ThreadPoolExecutor`
- 如何识别斜杠命令？  ***event.text strip 后做精确字符串匹配***（text in {"/new","/reset"}、text == "/help"），在进入 session/loop之前短路，不进 Loop）
- 构建注意事项：init 多个字段 包含session_lock、channel_dict，start，

init初始化有关的内容，message队列，session历史，agent对话loop，channel 记录dict，sessionkey的locks，bus的ThreadPoolExecutor。

start我们的gateway，核心就是启动和注册两个thread分别监督message bus的两个队列。

submit和submit_raw函数 分别根据bus传入对应的字段（解析完毕和未解析）

_inbound_dispatch_loop + _process_inbound 两个函数分别是线程while循环监督message bus的task函数 和 当线程监督到之后进行对应的调用agent回答函数的处理  

### 4. 如何对于同一个session key防止多个操作影响？

- 不管别的地方怎么做 gateway必须针对任何一个session key有唯一的一把锁（dict）
- 当我们线程提交给线程池某个监控到的message时 我们把处理函数一并传入参数 在处理函数中 首先获取key 判断key是否被锁住 锁住则等待（串行） 否则直接执行



### 5. Agent Loop的职责有哪些？

`核心 OTAP 循环：observe -> think -> act -> persist`，只吃 Session + Message，吐 TurnResult，不认识任何平台 SDK / 队列 / Channel。

- **Observe（观察）**：把传入的 `InboundEvent.text` 转成 `Message(role=user)` 追加进 `Session.messages`，维护 seq_id / message_count / 时间戳（`session.add_message`）。Loop 不解析平台原始载荷，那是 Channel 的活。
- **Think（思考）**：用 `ContextBuilder` 把 Session 组装成 OpenAI 格式的 `messages` 列表（system prompt + 多轮对话 + ephemeral 平台说明），调 `model.chat(messages)` 拿回复。多步骤时循环 think。
- **Act（行动 / 工具内循环）**：解析模型返回的 `tool_calls` -> 执行工具 -> 把结果作为 `Message(role=tool)` 回注 Session -> 再 think，直到模型不再请求工具或撞 `max_iterations` 上限。
- **TurnResult 产出**：返回 `final_text` / `session_id` / `completed` / `interrupted` / `error`，供 Gateway 回写。Loop 自己不出站，不调 Channel.deliver。
- **预算与中断**：`max_iterations` 限制工具循环次数防爆走；`request_interrupt()` 设 flag，每轮检查，被打断时优雅返回 `interrupted=True`。
- **Persist 委托**：Loop **不写磁盘**，只改内存里的 Session；落盘（transcript / index）交给 Gateway 调 SessionStore.save，保持 Loop 纯净可测试。
- **通道无关**：`ephemeral_prompt`（"你在飞书群里"）仅注入当次 API 不进 transcript；Loop 看不到 `session_key` / `chat_id`，多 Channel 隔离全靠 Gateway。
- **日志与容错**：`model.chat` 抛异常时 `logger.exception` 记录并返回 `error`，不向上抛穿；用户消息已入 session，由上层决定是否回滚。

一句话：Loop 是"会思考会调工具的纯函数"，平台差异和持久化都在它之外。




### 6. 进程和线程？

- 进程： 资源分配的基本单位 一段代码跑起来 系统为他创建一个进程 分配内存 打开文件、分配网络端口
  - 独立内存空间：彼此之间不通信就不能互享
  - 隔离性强：一个崩溃不会影响其他
  - 开销大：创建、切换进程的成本大
- 线程： cpu调度的基本单位，一个进程可以有多个线程
  - 共享内存：可以同时读写一份变量，通信快，但是需要加锁防止数据竞争
  - 开销小：创建和切换迅速
  - 一崩全崩：一个毁掉其他全部毁掉
